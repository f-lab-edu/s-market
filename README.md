## User

### 회원 가입

📌 상황
- 사용자는 이메일과 비밀번호를 입력해 회원가입을 요청
- 이메일이 중복되지 않았는지 확인한 후, 비밀번호를 해시화하여 저장 
- 회원가입이 완료되면 사용자에게 성공 메시지를 반환

📌 흐름
- 사용자가 POST /users/signup API 호출
- 이메일 중복 여부 확인
- 비밀번호를 해시화 후 DB 저장
- 회원가입 성공 응답 반환

⚠️ 예외 상황  
- 입력값이 부족한 경우
- 이미 가입된 이메일인 경우


### 로그인

📌 상황
- 사용자가 이메일과 비밀번호로 로그인
- 이메일이 존재하는지 확인한 후, 비밀번호를 검증
- 로그인 성공 시 JWT 토큰을 발급하여 반환

📌 흐름
- 사용자가 POST /users/login API 호출
- 이메일 존재 여부 확인
- 비밀번호 일치 여부 검증
- JWT 토큰 발급 후 응답 반환

⚠️ 예외 상황  
- 이메일이 존재하지 않는 경우
- 비밀번호가 틀린 경우


## Product

### 상품 목록 조회

📌 상황  
- 사용자는 쇼핑몰에서 판매 중인 상품 목록을 확인
- 사용자가 요청한 페이지의 상품 목록을 반환

📌 흐름  
- 사용자가 GET /products API 호출
- 상품 목록을 조회하여 반환


⚠️ 예외 상황  
- 상품이 하나도 없는 경우 빈 배열 반환  


### 상품 상세 조회

📌 상황 
- 사용자가 특정 상품의 상세 정보를 확인
- 상품의 정보(가격, 설명, 재고)를 반환

📌 흐름  
- 사용자가 GET /products/{id} API 호출
- 상품 정보를 조회하여 반환


⚠️ 예외 상황  
- 존재하지 않는 상품 조회하는 경우

## Cart

### 장바구니 추가  

📌 상황 
- 사용자가 원하는 상품을 장바구니에 추가
- 같은 상품이 이미 장바구니에 있다면 수량만 증가

📌 흐름  
- 사용자가 POST /cart API 호출 (상품 ID 및 수량 포함)
- 장바구니에 해당 상품이 있는지 확인
- 없으면 새로 추가
- 있으면 기존 수량 증가
- 장바구니 정보 저장 후 성공 응답 반환

⚠️ 예외 상황  
- 상품 수량이 0 이하일 경우

### 장바구니 조회

📌 상황
- 사용자가 장바구니에 담긴 상품 조회
- 사용자의 장바구니 정보를 반환

📌 흐름  
- 사용자가 GET /cart API 호출
- 장바구니 데이터를 조회하여 반환

⚠️ 예외 상황  
- 장바구니가 비어있는 경우

### 장바구니 목록 업데이트

📌 상황
- 사용자가 장바구니에 담긴 여러 상품의 수량이나 선택 옵션 등을 한 번에 업데이트하고자 함

📌 흐름
- 사용자가 PUT 또는 PATCH /cart API 호출 (업데이트할 상품 목록 및 각각의 변경사항 포함)
- 시스템이 요청으로 전달받은 각 상품의 ID를 기준으로 사용자의 장바구니 내 존재 여부 확인
- 존재하는 경우, 각 상품의 수량 또는 기타 업데이트 필드 수정
- 예: 수량 변경, 선택 옵션 변경 등
- 업데이트된 장바구니 정보 저장 후 성공 응답 반환

⚠️ 예외 상황
- 업데이트 요청 데이터에 포함된 상품 ID가 장바구니에 존재하지 않는 경우
- 예: 해당 상품이 이미 삭제되었거나 장바구니에 없는 경우
- 수정하려는 수량이 0 이하인 경우 (예외 처리 필요)
- 요청 형식 오류 또는 필수 값 누락 시 에러 응답 반환

### 장바구니 특정 상품 삭제

📌 상황
- 사용자가 장바구니에 담긴 특정 상품을 삭제

📌 흐름
- 사용자가 DELETE /cart/{productId} API 호출
- 장바구니에 해당 아이템이 존재하는지 확인
- 장바구니에서 해당 상품 삭제
- 성공 응답 반환

⚠️ 예외 상황
- productId가 존재하지 않는 경우
- 사용자의 장바구니가 비어 있는 경우

### 장바구니를 한번에 삭제

📌 상황
- 사용자가 장바구니의 모든 상품을 한 번에 삭제

📌 흐름
- 사용자가 DELETE /cart API 호출
- 시스템이 사용자의 장바구니에 있는 모든 상품을 조회
- 장바구니 비우기 (전체 삭제)
- 성공 응답 반환

⚠️ 예외 상황
- 장바구니가 이미 비어 있는 경우

## Order

### 주문 생성

📌 상황
- 사용자가 장바구니에 담긴 상품을 주문
- 결제가 완료되면 주문 확정

📌 흐름
- 사용자가 POST /orders API 호출
- 시스템이 장바구니 데이터를 기반으로 주문 생성
- 주문 상태를 "결제 대기"로 설정
- 주문 생성 완료 후 성공 응답 반환

⚠️ 예외 상황  
- 장바구니가 비어 있을 경우
- 재고가 부족한 상품이 포함된 경우

### 주문 수정

📌 상황
- 사용자가 주문 상세 정보(배송지, 연락처, 일부 상품 옵션 등)를 수정하고자 함
- 단, 주문 수정은 결제 대기 상태 등 수정이 허용된 주문에 한해서만 가능


📌 흐름
- 사용자가 PUT 또는 PATCH /orders/{id} API 호출 (수정할 주문 ID와 변경할 데이터 포함)
- 시스템이 주문 상태를 확인하여 수정 가능 여부 판단
- 예: 주문 상태가 "결제 대기"인 경우에만 수정 가능
- 수정 가능한 경우, 주문 상세 정보 업데이트 처리
- 수정된 주문 정보 저장 후 성공 응답 반환

⚠️ 예외 상황
- 주문 상태가 이미 결제 완료, 배송 중 또는 완료된 주문인 경우
- 수정 요청 데이터에 필수 정보 누락 또는 형식 오류 발생 시 에러 응답 반환

### 주문 조회

📌 상황
- 사용자가 자신의 주문 목록을 확인
- 사용자의 주문 목록을 반환

📌 흐름
- 사용자가 GET /orders API 호출
- 시스템이 사용자 주문 목록을 조회하여 반환

⚠️ 예외 상황  
- 주문 목록이 비어있는 경우

### 주문 취소

📌 상황
- 사용자가 주문을 취소
- 결제 완료 전이면 취소가 가능

📌 흐름
- 사용자가 POST /orders/{id}/cancel API 호출
- 시스템이 주문 상태 확인
- 결제 완료 이전이면 주문을 "취소됨" 상태로 변경
- 재고 복구 후 성공 응답 반환

⚠️ 예외 상황 
- 이미 결제 완료된 주문
- 배송 중인 주문

## Payment

### 결제

📌 상황
- 사용자가 주문한 상품을 결제
- 결제 시뮬레이션

📌 흐름
- 사용자가 POST /payments/{orderId} API 호출
- 결제 요청
- 결제 성공이면 결제 완료, 실패면 결제 실패로 주문 상태 변경

⚠️ 예외 상황
- 주문이 이미 결제된 경우

### 결제 취소

📌 상황  
- 사용자가 결제를 취소
- 결제 취소가 성공하면 주문 상태도 변경

📌 흐름  
- 사용자가 POST /payments/{orderId}/cancel API 호출
- 주문 상태 확인
- 결제 취소 가능하면 상태 변경 후 성공 응답 반환

### 적용해보고 싶은 기술

📌 서비스 분리 
- User Service: 회원가입, 로그인, JWT 발급/갱신, 사용자 정보 관리
- Product Service: 상품 목록 조회, 상품 상세 조회, 검색 기능
- Cart Service: 장바구니 추가, 수정, 조회, 삭제
- Order Service: 주문 생성, 수정, 조회, 취소
- Payment Service: 결제 처리, 결제 취소

📌 DB 선택  
RDBMS (MySQL) 사용  
Cart → (Redis) 사용

📌 DB 아키텍처  
Master-Slave Replication: 읽기/쓰기 분리
Connection Pool설정

📌 비동기 이벤트 처리  
- Kafka로 주문 생성 → 결제 → 재고 감소 이벤트 기반 처리

📌 인증 / 보안
- JWT 기반 인증 적용
- Access Token (만료 짧음) + Refresh Token (길게 유지)
- OAuth2 지원 (Google)
- BCrypt 비밀번호 암호화
- SQL Injection 방지
- Rate Limiting DDoS 방어

📌 CI/CD  
- github action

📌 트래픽 처리  
- 로드 밸런싱

📌 모니터링 및 성능 분석  
- Prometheus 모니터링 시스템
- Grafana
